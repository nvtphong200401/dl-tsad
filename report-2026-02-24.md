# Running Report: `phase2_stl_llm.yaml` (2026-02-24)

This document explains exactly the pipeline executed by:

```bash
python experiments/run_benchmark.py --configs phase2_stl_llm --categories trend
```

Configuration used (`configs/pipelines/phase2_stl_llm.yaml`):
- Step 1: `STLProcessor` with `window_size=100`, `stride=10`, `forecast_horizon=64`, `robust=true`
- Step 2: `EvidenceBasedDetection` with categories:
  - `forecast_based`
  - `statistical_tests`
  - `distribution_based`
  - `pattern_based`
- Step 3: `LLMReasoningScoring` with `batch_size=10`, `pre_filter_percentile=90`
- Step 4: `F1OptimalThreshold`, `min_anomaly_length=3`, `merge_gap=5`

---

## 1) Input and shape contracts for this run

### Raw inputs
- `X_train`: training series, shape `(T_train, D)`
- `X_test`: test series, shape `(T_test, D)`
- `y_test`: labels for threshold tuning (required because this config uses `F1OptimalThreshold`)

If the signal is univariate, the pipeline internally uses `D=1` and works with `(T, 1)`.

### Windowing used in this config
With `W=100` and `S=10`:

$$
N = \left\lfloor\frac{T - W}{S}\right\rfloor + 1
$$

Step-1 output shape is `(N, 100, D)`.

---

## 2) Step 1 in this run: `STLProcessor`

`STLProcessor` does three important jobs:

1. **Normalize** windows using train mean/std.
2. **Fit STL baseline** on reconstructed training signal (trend + seasonal pattern).
3. **Generate per-window expected behavior** on test windows for Step-2 evidence.

### What it computes during `fit_transform(X_train)`
- training mean/std from flattened windows
- training quantiles (`P01..P99`), min, max
- seasonal period (provided or auto-detected)
- trend line parameters (`slope`, `intercept`)
- residual error distribution (used for synthetic uncertainty samples)

### What it computes during `transform(X_test)`
- normalized test windows `(N,100,D)`
- one forecast package per window:
  - `forecast`: expected in-window normalized values
  - `quantiles`: `P01, P10, P25, P50, P75, P90, P99`
  - `samples`: synthetic trajectories from residual bootstrap
  - `uncertainty`: `P90 - P10`

This forecast context is passed to Step 2 via `set_forecast_context()`.

---

## 3) Step 2 in this run: `EvidenceBasedDetection` (detailed)

This is the core detector in `phase2_stl_llm`. It does not directly threshold anomalies. It builds a **structured evidence profile per window**, then compresses that profile into a scalar anomaly score.

### 3.1 Inputs received by Step 2
For each test window index `i`, detector uses:
- normalized window values: `window = X_processed[i, :, 0]`
- STL forecast package for the same window (`forecast`, quantiles, uncertainty)
- training statistics from Step 1 (`mean`, `std`, quantiles)
- training data sample (for baseline normalization)

So every score is contextualized by both:
- *expected behavior* (STL forecast)
- *normal historical behavior* (training stats/sample)

### 3.2 Evidence extraction categories enabled in this config

#### A) `forecast_based`
Compares actual window against STL expected values/intervals.
Typical signals include forecast error and uncertainty violations such as:
- mean absolute error-like magnitude
- mean squared error-like magnitude
- ratio of points violating expected quantile interval
- average surprise relative to uncertainty

This catches contextual anomalies where values are not globally extreme but are inconsistent with expected local pattern.

#### B) `statistical_tests`
Tests local deviation intensity relative to train baseline, such as:
- max absolute z-score in window
- Grubbs-style outlier statistic
- CUSUM-like cumulative shift

This catches abrupt local shifts/spikes even if forecast context is imperfect.

#### C) `distribution_based`
Compares the window distribution to normal reference distribution:
- KL-divergence-like discrepancy
- normalized Wasserstein distance

This catches shape changes (distribution drift) beyond single-point outliers.

#### D) `pattern_based`
Checks temporal structure mismatch:
- autocorrelation pattern differences
- volatility ratio mismatch
- slope/trend mismatch

This catches anomalies that preserve mean level but alter dynamics.

### 3.3 Baseline fitting inside detector (`fit`)
Before scoring test windows, detector builds internal normalization baselines from training windows:

1. samples training values (`train_data_sample`)
2. computes per-window baseline metrics on sampled train windows
3. stores metric percentiles, especially `p95`, per metric

These `p95` baselines are used to normalize evidence magnitudes at inference time, so heterogeneous metrics become comparable.

### 3.4 Per-window evidence to scalar score
For each window, detector runs:

1. `extractor.extract(...)` → evidence dictionary
2. attach `window_index`
3. aggregate evidence with weighted normalized sum

Aggregation logic:
- for each configured metric with weight `w_m`:
  - read raw value `v_m`
  - normalize by training `p95_m` (if available):

$$
	ilde{v}_m = \frac{v_m}{p95_m}
$$

  - clip normalized value to `[0,10]`
  - add weighted contribution `w_m * \tilde{v}_m`
- divide by total active weight

Final window score:

$$
score_i = \frac{\sum_m w_m \cdot clip(\tilde{v}_m, 0, 10)}{\sum_m w_m}
$$

Output of Step 2:
- `subsequence_scores`: shape `(N,)`
- `evidence_results`: list of detailed per-window evidence dicts

### 3.5 Why this detector helps in `phase2_stl_llm`
- STL gives fast expected baseline without heavy FM inference.
- Multi-category evidence captures different failure modes simultaneously.
- p95-based normalization stabilizes metric scales.
- Rich evidence is reusable by LLM stage, not discarded.

---

## 4) Step 3 in this run: `LLMReasoningScoring`

This stage refines Step-2 scores with selective LLM calls.

### 4.1 Pre-filter policy (`pre_filter_percentile=90`)
- Compute P90 of window scores.
- Only top 10% windows (or at least 1 window) are sent to LLM.

### 4.2 LLM scoring behavior
- LLM receives suspicious windows + their evidence.
- Also receives a few low-score windows as normal contrast.
- LLM returns confidence-like anomaly scores for suspicious windows.

### 4.3 Merge strategy
- suspicious windows → use LLM scores
- non-suspicious windows → keep normalized statistical score scaled low (`*0.3`)

Then convert `(N,)` window scores to `(T,)` point scores using overlap averaging.

---

## 5) Step 4 in this run: `F1OptimalThreshold` + cleanup

### 5.1 Threshold search
Given `point_scores` and `y_test`, Step 4 scans percentiles from 80 to 99.9 and chooses the threshold that maximizes F1.

### 5.2 Binary prediction and segment refinement
1. `pred = (point_scores >= threshold)`
2. remove anomaly segments shorter than 3 points (`min_anomaly_length=3`)
3. merge adjacent anomaly segments with gap <= 5 (`merge_gap=5`)

Final output is cleaner event segments, not only point spikes.

---

## 6) Final output object from this run

Pipeline returns `PipelineResult` with:
- `predictions`: `(T_test,)` binary anomaly labels
- `point_scores`: `(T_test,)` continuous anomaly score
- `subsequence_scores`: `(N_test,)` evidence-based window scores
- `threshold`: scalar selected by F1 search
- `metadata`: component names, window settings, evidence presence/count
- `execution_time`: per-step timings + total

---

## 7) One concrete shape walkthrough

If `X_test` is `(1000,1)` with `W=100,S=10`:

$$
N = \left\lfloor\frac{1000-100}{10}\right\rfloor + 1 = 91
$$

- Step 1 (`STLProcessor`) output: `(91,100,1)` + 91 forecast/evidence contexts
- Step 2 (`EvidenceBasedDetection`) output: `subsequence_scores (91,)` + 91 evidence dicts
- Step 3 (`LLMReasoningScoring`) output: `point_scores (1000,)`
- Step 4 (`F1OptimalThreshold`) output: `predictions (1000,)`, `threshold`

---

## 8) Short operational interpretation

- High `subsequence_scores[i]`: window `i` has strong multi-signal evidence of abnormality.
- High `point_scores[t]`: timestamp `t` is repeatedly covered by high-risk windows.
- `predictions[t]=1`: timestamp marked anomalous after supervised thresholding and segment cleanup.

In this config, **EvidenceBasedDetection is the primary anomaly signal generator**, and the LLM stage is a selective refinement layer on top.
